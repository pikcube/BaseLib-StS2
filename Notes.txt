Notes

Async State Machine generation/await calls

State Machine -
integer "state", starting from -1
field for each method parameter
field for result of each await (if stored in a local var in method)

each awaited call is another state
one TaskAwaiter<T> field for each await

StateMachine MoveNext method - contains basically all of actual method
Starts by checking state and jumping based on it (simpler state machine may simply use an if statement)

Each state is a branch with 3.5 sections.
 1 When state is initially reached (comes from ending of previous state)
    - Call async method of state (returns task) and get awaiter from task
    - If awaiter is done, skip to section 4. Otherwise, continue to section 2
 2 Set up AsyncTaskMethodBuilder.AwaitUnsafeOnCompleted
 3 Branch destination when MoveNext is called again after done awaiting.
    - Sets up so that section 4 will work appropriately same as coming from section 1
 4 Finishing section of state AND starting section of next state.
    - Calls TaskAwaiter.GetResult
    - Intermediate non-awaited code will also occur here.
    - (section 1 of next state)

State is only set if it needs to wait, in which case the method returns and waits for next execution (when current awaiter ends).
When initially reaching a state, the current state will be -1.





await call - 
The "await" specifically just *waits*. The actual async startup is performed by the method call.
The same task awaited multiple times will only execute once.
Async method call -
new instance of state machine class (store in local)
AsyncTaskMethodBuilder.Create for builder instance, store in state machine class field
store method call parameters in state machine class fields
set state machine "state" int to -1
start state machine by getting builder from state machine local instance and calling AsyncMethodTaskBuilder.Start
get builder from state machine and return builder.Task (property)

CARDS - 
ALL cards in combat are assumed to be in a pile (CardPile). If they are not in a pile, they will be treated as not in combat (no damage calc and similar).
What pile a card is in is checked inefficiently - iterate through all of owner's piles and return first contains result.
This means there is a *lot* of iteration of all cards in all piles. Might be a problem if someone has like a million cards.
